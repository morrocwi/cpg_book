# CPG Book

**ห้องสมุดความรู้ของ Yaoharee Lahtee และ Walancha**

`cpg_book` คือพื้นที่กลางสำหรับรวบรวม จัดหมวดหมู่ เชื่อมโยง และพัฒนาความรู้ของ **Yaoharee Lahtee** และ **Walancha** ให้สามารถกลับมาอ่าน ตรวจสอบ อ้างอิง และต่อยอดร่วมกับมนุษย์และ AI ได้อย่างมีบริบท

ห้องสมุดนี้ไม่ได้มีไว้เก็บเฉพาะหนังสือ แต่รวมถึงความรู้ที่เกิดจากการอ่าน การทำงาน ประสบการณ์ การสนทนา การวิจัย และการสะท้อนคิดร่วมกัน

---

## 1. เป้าหมายของห้องสมุด

- รักษาความรู้และประสบการณ์ไม่ให้สูญหายไปกับบทสนทนาหรือเหตุการณ์เฉพาะครั้ง
- เปลี่ยนความรู้ฝังลึกให้เป็นเนื้อหาที่อ่าน ตรวจสอบ และพัฒนาต่อได้
- เชื่อมหนังสือ แหล่งอ้างอิง ประสบการณ์ หลักคิด และคำถามที่เกี่ยวข้องเข้าด้วยกัน
- เป็นพื้นที่กลางระหว่าง Yaoharee, Walancha และ AI หลายตัวที่ทำงานร่วมกับทั้งสองคน
- สนับสนุนการพัฒนาเป็นบทความ หนังสือ หลักสูตร งานวิจัย ระบบความรู้ หรือระบบ AI ในอนาคต
- รักษาประวัติการเปลี่ยนแปลงของความคิดโดยไม่ทำให้ข้อสรุปเดิมกลายเป็นสิ่งตายตัว

---

## 2. เจ้าของบริบทและความหมาย

ห้องสมุดนี้เป็นของ:

1. **Yaoharee Lahtee**
2. **Walancha**

ทั้งสองคนเป็นเจ้าของประสบการณ์ บริบท และความหมายที่ถูกนำมาสกัดเป็นองค์ความรู้ใน repository นี้

AI สามารถช่วย:

- ค้นหา
- สรุป
- จัดหมวดหมู่
- เชื่อมโยงแนวคิด
- ตั้งคำถาม
- เสนอการตีความ
- ตรวจความสอดคล้อง
- ช่วยพัฒนาเนื้อหา

แต่ AI:

- ไม่ใช่เจ้าของประสบการณ์
- ไม่ใช่ตัวแทนสมบูรณ์ของบุคคล
- ไม่ควรเปลี่ยนข้อเสนอชั่วคราวให้เป็นความจริงโดยไม่มีการตรวจสอบ
- ไม่ควรอ้างเจตนาของ Yaoharee หรือ Walancha หากไม่ได้รับการยืนยัน

```text
Human experience and judgment
            ↓
        CPG Book
            ↓
shared and reviewable knowledge
            ↓
humans and multiple AI systems
```

## 3. หลักการของห้องสมุด

### 3.1 เอกสารไม่เท่ากับตัวบุคคล

ไฟล์ บันทึก หรือแบบจำลองใด ๆ เป็นเพียงตัวแทนบางส่วนของความคิด ณ เวลาหนึ่ง ไม่ใช่ตัวบุคคลทั้งหมด

```text
Documented knowledge ≠ complete person
```

### 3.2 บทสรุปไม่เท่ากับแหล่งต้นฉบับ

บันทึกการอ่านและบทสรุปควรเชื่อมกลับไปยังต้นทางเมื่อทำได้ และต้องแยกคำกล่าวของแหล่งข้อมูลออกจากการตีความของผู้บันทึก

```text
Summary ≠ source
```

### 3.3 ความมั่นใจไม่เท่ากับหลักฐาน

การเขียนได้อย่างมั่นใจหรือสอดคล้อง ไม่ได้ทำให้ข้อสรุปนั้นถูกต้องโดยอัตโนมัติ

```text
Confidence ≠ evidence
```

### 3.4 ประสบการณ์ไม่เท่ากับกฎทั่วไป

ประสบการณ์ส่วนบุคคลสามารถเป็นต้นทางของบทเรียนหรือสมมติฐานได้ แต่ต้องไม่ถูกยกระดับเป็นกฎสากลโดยไม่มีการทบทวน

```text
Experience ≠ universal rule
```

### 3.5 ความรู้ต้องรักษาบริบท

รายการความรู้ควรระบุว่าเกิดขึ้นเมื่อใด มาจากไหน อยู่ในบริบทใด ใช้เพื่ออะไร และมีขอบเขตเพียงใด

### 3.6 ความรู้ต้องแก้ไขได้

เมื่อพบข้อมูลใหม่ ความเข้าใจที่ดีขึ้น หรือข้อผิดพลาด ต้องสามารถปรับปรุง พร้อมรักษาประวัติที่จำเป็นต่อการเข้าใจพัฒนาการของความคิด

### 3.7 การตีความของ AI ไม่เท่ากับเจตนาของเจ้าของ

```text
AI interpretation ≠ owner intention
```

## 4. ขอบเขตเนื้อหา

เนื้อหาที่เหมาะกับ `cpg_book` ได้แก่:

- หนังสือ บทความ งานวิจัย และเอกสารอ้างอิง
- บันทึกการอ่านและข้อสังเกต
- แนวคิด ปรัชญา และกรอบการใช้เหตุผล
- Personal Reasoning Foundation
- บทเรียนจากประสบการณ์จริง
- ความรู้ด้านครอบครัว ศาสนา สังคม วัฒนธรรม ธุรกิจ และการทำงาน
- คำศัพท์ นิยาม และแผนที่แนวคิด
- ร่างบทความ หนังสือ หลักสูตร หรือองค์ความรู้
- ความรู้ร่วมของ Yaoharee และ Walancha
- เนื้อหาที่เตรียมไว้สำหรับพัฒนาต่อในระบบ AI หรือระบบองค์กร

สิ่งที่ไม่ควรเก็บโดยตรง:

- รหัสผ่าน กุญแจ API หรือข้อมูลลับ
- ข้อมูลลูกค้าหรือข้อมูลส่วนบุคคลที่ไม่จำเป็น
- เนื้อหาลิขสิทธิ์ฉบับเต็มที่ไม่มีสิทธิในการจัดเก็บหรือเผยแพร่
- บันทึกปฏิบัติการรายวันที่ไม่มีคุณค่าทางความรู้ระยะยาว
- คำสั่ง workflow หรือ skill ที่มีหน้าที่สั่งการระบบโดยตรง
- ข้อสรุปที่อ้างเป็นข้อเท็จจริงโดยไม่มีแหล่งที่มา

## 5. ความแตกต่างจาก repository ประเภทอื่น

`cpg_book` เป็น **library and knowledge repository**

มันไม่ใช่:

- workflow engine
- operational database
- task tracker
- skill registry
- solver
- production control system

```text
CPG Book      preserves knowledge, notes, concepts, and shared meaning
CPG EQ        registers equations and formal relations
CPG Solver    computes or tests formal models
Skill systems store bounded capabilities
Workflow      defines operational steps and state transitions
```

เนื้อหาจากห้องสมุดสามารถถูกนำไปพัฒนาต่อเป็นงานวิจัย สมการ skill workflow บทความ หรือระบบ AI ได้ แต่การนำไปใช้ต้องอยู่ภายใต้กฎของระบบปลายทางนั้นเอง

## 7. รูปแบบรายการความรู้

รายการหนึ่งควรมี metadata อย่างน้อย:

```yaml
id: BOOK-YYYY-NNN
title: "ชื่อรายการ"
type: book_note
language: th
owners:
  - Yaoharee Lahtee
  - Walancha
contributors: []
source:
  title: "ชื่อแหล่งต้นทาง"
  author: null
  url: null
  published_at: null
created_at: YYYY-MM-DD
updated_at: YYYY-MM-DD
status: draft
claim_state: interpretive_note
tags: []
summary: >
  สรุปสาระสำคัญโดยไม่แทนที่แหล่งต้นฉบับ
owner_reflection: >
  ความหมายหรือบทเรียนที่เจ้าของห้องสมุดมองเห็น
open_questions: []
related_items: []
```

สถานะที่แนะนำ:

```text
draft        บันทึกเริ่มต้น
reviewed     ผ่านการทบทวน
shared       ทั้งสองคนร่วมรับรู้หรือร่วมพัฒนา
reference    พร้อมใช้เป็นแหล่งอ้างอิงภายใน
superseded   มีรายการใหม่แทนที่
archived     เก็บเพื่อประวัติ
```

## 8. วิธีเพิ่มความรู้

```text
Source or experience
        ↓
Record provenance and context
        ↓
Separate source, summary, and reflection
        ↓
Add uncertainty and open questions
        ↓
Review by the relevant owner
        ↓
Index and connect related knowledge
        ↓
Revise when new evidence appears
```

หลักสำคัญคือ:

> เก็บที่มาก่อนเพิ่มการตีความ และแยกการตีความออกจากต้นฉบับ

## 9. การใช้ร่วมกับ AI หลายตัว

AI แต่ละตัวสามารถอ่านส่วนที่เกี่ยวข้องกับโดเมนของตน เช่น:

- HR AI อ่านหลักเกี่ยวกับคน ศักดิ์ศรี บทบาท และการเรียนรู้
- COO AI อ่านหลักเกี่ยวกับการตัดสิน ทรัพยากร ผลกระทบ และการแก้ไข
- Imam AI อ่านกรอบศาสนา จริยธรรม หลักฐาน และขอบเขตอำนาจ
- Research AI อ่านแหล่งอ้างอิง สมมติฐาน ข้อโต้แย้ง และคำถามเปิด
- Writing AI อ่านแนวคิด โครงสร้าง และวิธีอธิบาย

AI แต่ละตัวไม่จำเป็นต้องให้ข้อสรุปเหมือนกัน แต่ควรแยกให้ชัดระหว่าง:

```text
source
summary
interpretation
personal reflection
provisional claim
verified claim
```

## 10. การดูแลและพัฒนาต่อ

- ใช้ ID ที่คงที่สำหรับรายการความรู้
- ระบุเจ้าของบริบทให้ชัด
- แยกความรู้เฉพาะบุคคลออกจากความรู้ร่วม
- การแก้ไขสาระสำคัญควรมีเหตุผลหรือ changelog
- ไม่ลบประวัติที่จำเป็นต่อการเข้าใจพัฒนาการของแนวคิด
- รายการที่ล้าสมัยควรใช้สถานะ `superseded` หรือ `archived`
- เนื้อหาที่เป็นความรู้ร่วมควรอยู่ใต้ `people/shared/` หรือระบุเจ้าของทั้งสองคน
